The LSP diagnostics system appears well-architected but has several potential issues that
could prevent it from working properly for the current project. The main issue seems to be
related to project structure and LSP server initialization.

start, preventing diagnostics for TypeScript files.

#### Long-term Improvements:

1. Retry Mechanism: Implement exponential backoff for failed LSP servers
2. Better Error Messages: Provide user-friendly error messages when LSP fails
3. Configuration Validation: Validate LSP configuration before attempting to spawn servers
4. Monorepo Support: Explicitly handle monorepo structures in root detection

### Root Cause Analysis

The most likely reason the LSP diagnostics aren't working for the current project is:

1. Monorepo Structure: The LSP system expects a single package structure but is operating
within a monorepo
2. TypeScript Server Resolution: typescript-language-server or tsserver.js cannot be properly
resolved from the current directory
3. Project Root Mismatch: The LSP server might be initializing with packages/opencode/ as
root instead of the repository root
