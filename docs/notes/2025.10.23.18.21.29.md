Title: Token usage accounting cross-UI diagnostic – plan, findings,
and recommendations

Date: 2025-10-23

Executive summary

• Two interfaces report token usage differently: Web UI aggregates
tokens across all assistant messages in a session; TUI reports tokens
for the current/last interaction (with compaction). Both reflect
aspects of real usage but diverge due to prompt assembly, history
handling, and UI goals.
• The discrepancy arises from how the session context is constructed
and how token costs are calculated and surfaced. The Web view aligns
with total input tokens sent to the provider; the TUI view reflects
the per-turn marginal tokens after compaction.
• A single source of truth is preferred for budgeting and UX,
requiring alignment of both token counting and context window
accounting across the stack (frontend Web, TUI, and backend prompt
assembly).

Objectives

• Determine how many tokens are actually sent to the server per turn
and per session.
• Understand provider cost calculation for tokens and whether cached
tokens are correctly handled.
• Assess correctness of TUI accounting vs Web accounting; determine
which is right or how to harmonize.
• Evaluate whether this is primarily a UX issue or a deeper
data-model/implementation problem.
• Propose a canonical model for token/cost metrics and a path to unify
displays.

Key findings

• Context scope: Web accumulates tokens across all assistant messages;
TUI appears to count only the last message in the current context
(compaction-limited).
• Token types: input, output, reasoning, and cache interactions
(read/write) exist; their mapping to provider billing depends on the
actual prompt payload.
• Data flow: share_data endpoint returns per-session data; UI
components derive tokens from data stored per session. The current
code paths indicate session-scoped data rather than cross-session
aggregation.
• Likely misalignment: The TUI display is underrepresenting total
session token usage, while Web shows the total tokens sent to the
provider across the session length.

Canonical model and recommended alignment

• Define canonical metrics:
 • total_session_input_tokens: sum of input tokens across all prompts
 sent to the provider for the session so far.
 • total_session_cost: cost billed by provider for the session so far
 (derived from total_session_input_tokens and token pricing, plus any
 output tokens and overhead).
 • context_tokens_for_next_call: tokens that will be included in the
 next provider prompt (current context window size for the upcoming
 turn).
 • per_turn_deltas: tokens used for the last turn (optional
 drill-down).
• Align UI components to canonical metrics:
 • Web UI: display total_session_input_tokens and total_session_cost;
 optionally show per-turn deltas for transparency.
 • TUI: switch to accumulating per-turn tokens to reflect total
 session usage (i.e., compute total_session_input_tokens similarly)
 and expose a delta per turn to show how much the next prompt will
 consume.
• Handle Summary:
 • If the prompt payload used by the model uses summaries, ensure
 total_session_input_tokens reflect the actual payload length that was
 sent (i.e., if a summary replaced prior content in the prompt, only
 the actual transmitted content should count toward
 total_session_input_tokens).
 • If summaries are purely display-only, decide on whether to treat
 them as compression that reduces context size for the next call;
 ensure the canonical metrics track the actual used tokens.


Implementation plan (high level)

• Step 1: Instrumentation
 • Add instrumentation to record tokens per prompt construction:
  • Log per-turn input tokens, output tokens, and any cache
  interactions used in building the prompt.
  • Record the actual payload length that is sent to the LLM per turn.
 • Add a small reconciliation layer that computes:
  • total_session_input_tokens = sum(all prompt payload lengths sent
  so far)
  • context_tokens_for_next_call = length of the current prompt
  payload (or a truncated/condensed payload if applicable)

• Step 2: Refactor UI accounting
 • Modify tui internal token calculation to accumulate across all
 messages actually sent to the model (including context relevant
 messages and summaries as used in the prompt payload).
 • Modify web to continue its accumulation but ensure it uses a single
 canonical source for total_session_input_tokens that can be
 cross-checked.
• Step 3: Tests
 • Unit tests for canonical metrics calculation.
 • Integration tests simulating multi-turn sessions, with and without
 summaries, to verify that UI metrics align with the canonical metrics.
• Step 4: UX and docs
 • Introduce a unified metrics panel:
  • Total session input tokens
  • Total session cost
  • Context for next call (tokens in the upcoming prompt)
 • Documentation: add a glossary and migration guide; document
 definitions and edge cases (summaries, caching).


Risks and mitigation

• Risk: UX drift during migration (two numbers diverge in the
meantime)
 • Mitigation: Run a toggle during rollout; add inline explanations
 and a migration note.
• Risk: Performance overhead of instrumentation
 • Mitigation: Use lightweight counters and batch updates; ensure
 minimal impact on response times.
• Risk: Complexity of mapping cache tokens to actual billing
 • Mitigation: Start with straightforward mapping; refine as we learn
 more.


Deliverables

• A patch proposal with:
 • Code changes to tui and web to compute total_session_input_tokens
 consistently.
 • A canonical metrics module with tests.
 • Updated docs at docs/reports and docs/pages describing the
 canonical model.
 • PR outline with rationale and upgrade path.
