# Security Report: Seroval RCE / Prototype Pollution

### Summary

Unsafe deserialization in `seroval` (`<=1.4.0`, CVE-2026-23737 / GHSA-3rxj-6cgf-8cfw) is reachable in this project through SolidStart server functions. Attacker-controlled JSON sent to `/_server` is deserialized by `fromJSON()` before business logic runs. In local validation, this reliably produced deserialization faults and stack-overflow DoS behavior; upstream advisory documents an RCE path with additional exploitation steps.

### Details

The project relies on `@solidjs/start`, which pulls `seroval@1.3.2` transitively.

Reachability is explicit in framework routing:

- `node_modules/@solidjs/start/src/server/handler.ts:21` defines `SERVER_FN_BASE = "/_server"`.
- `node_modules/@solidjs/start/src/server/handler.ts:37` routes requests with that prefix to `handleServerFunction(...)`.

When a client invokes a SolidStart server action or query, the framework handles the request in `server-functions-handler.ts`. For `POST` requests, it passes the raw JSON body directly into `seroval.fromJSON`:

```typescript
// node_modules/@solidjs/start/src/server/server-functions-handler.ts
parsed = fromJSON(await event.request.json(), { plugins: [...] });
```

This occurs at `node_modules/@solidjs/start/src/server/server-functions-handler.ts:133`.

In development mode, server function IDs are discoverable from rendered output and decode to file/export pairs (generated by the TanStack server function plugin), which makes practical targeting straightforward.

`seroval@1.3.2` deserialization lacks the hardening introduced in `1.4.1` (depth limits, malformed node guards, key/symbol safety checks). As a result, crafted JSON can trigger deserialization error paths and deep-recursion DoS.

Because `packages/console/app` and `packages/enterprise` heavily utilize `"use server"` RPC endpoints (e.g., in `packages/console/app/src/routes/workspace/common.tsx`), these applications expose the vulnerable deserializer to the public internet on the `/_server` routing boundary.

### Impact

**Type:** Unsafe deserialization with prototype-manipulation and DoS impact; upstream advisory also documents an RCE path under additional preconditions.
**CVSS:** 7.5 (CVSS:3.1/AV:N/AC:H/PR:L/UI:N/S:U/C:H/I:H/A:H)
**CWE:** CWE-502 (Deserialization of Untrusted Data)

**Who is impacted:** Deployments of `@opencode-ai/console-app` and `@opencode-ai/enterprise` that expose SolidStart server functions. The CLI package (`@opencode-ai/opencode`) is not hosting this HTTP surface and is not directly exposed to this remote vector.

**Remediation:** Update `seroval` to `>= 1.4.1` (and ideally update `@solidjs/start` and `solid-js` to their latest patch versions to ensure compatibility with the updated serializer).

### Code Injection Investigation (Local)

Additional local testing was performed to determine whether arbitrary code execution can be directly triggered in this repository's current server-function flows.

Observed:

- Crafted payloads can reach deserialization and produce controlled parser failures (`SerovalDeserializationError`, `Unsupported node type`).
- Legacy `seroval@1.3.2` can deserialize inherited prototype members for constant/symbol selectors (for example `constructor`, `toString`, `__defineGetter__`), confirming unsafe primitive exposure in deserialization logic.
- In direct canary tests, no payload caused immediate execution of attacker-provided JavaScript (`globalThis` canary remained unchanged).
- Prototype manipulation was observed on returned objects, but no global `Object.prototype` pollution was observed in tested paths.

Assessment:

- This repository has confirmed exposure to unsafe deserialization and deserialization-abuse DoS behavior.
- Upstream still classifies this as potential RCE under additional preconditions (multi-request chain plus app-specific runtime usage of deserialized data).
- In this codebase, no direct one-request code-injection/RCE chain has been reproduced during local testing.

### Differential Validation Against Patched Version

To verify this is not a false positive, behavior was compared side-by-side between `seroval@1.3.2` (vulnerable) and `seroval@1.4.1` (patched) in an isolated local workspace.

Observed:

- `fromJSON({ t: { t: 2, s: "constructor" }, ... })`
  - `1.3.2`: returns `function Object`
  - `1.4.1`: rejects as malformed node (`specific: 8`)
- `fromJSON` with error selector `s: "constructor"`
  - `1.3.2`: accepts and returns coerced object
  - `1.4.1`: rejects as malformed node (`specific: 8`)
- Equivalent `fromCrossJSON` payloads show the same vulnerable vs patched split.

Interpretation:

- The vulnerable version accepts unsafe selector values via inherited property lookup during deserialization.
- The patched version enforces strict key validation and rejects those payloads.
- This differential confirms the vulnerability class is real and reachable in affected versions, even when a full app-specific RCE gadget chain is not demonstrated in this repository.
